## 1. 架构总览

### 1.1 应用类型选择

| 编号 | 名称 | Dify 应用类型 | 调用方式 | 原因 |
|------|------|-------------|---------|------|
| WF1 | 法规智能检索 | **Workflow** | `/v1/workflows/run` | 单次问答，无需多轮对话 |
| WF2 | 体系文件审核 | **Workflow** | `/v1/workflows/run` | 单次处理，多输入参数 |
| WF3 | 文案合规审核 | **Workflow** | `/v1/workflows/run` | 单次审核，结构化输出 |
| WF4 | 文案智能生成 | **Workflow** | `/v1/workflows/run` | 单次生成，多输入参数 |
| WF5 | 体系文件智能编制 | **Workflow** | `/v1/workflows/run` | 根据需求生成体系文件 |
| APP1 | 知识问答 | **Chatflow** | `/v1/chat-messages` | 需要多轮对话 + Memory |

### 1.2 模型选择建议

| 用途 | 推荐模型 | 备选 | 说明 |
|------|---------|------|------|
| LLM 主推理 | GPT-4o | Claude 3.5 Sonnet | 中文能力强，性价比高 |
| 问题分类 | GPT-4o-mini | GPT-3.5-turbo | 分类任务轻量即可 |
| Embedding | text-embedding-3-small | bge-large-zh | 知识库向量化 |
| Rerank | bge-reranker-v2-m3 | cohere-rerank | 提升检索精度 |

### 1.3 ���后端接口契约（已实现）

```
前端 → FastAPI(/api/workflow/run) → Dify Cloud(/v1/workflows/run)
前端 → FastAPI(/api/workflow/chat) → Dify Cloud(/v1/chat-messages)
```

每个 Workflow 有独立的 API Key，通过 `.env` 配置：
```
DIFY_WF_REGULATION_SEARCH_KEY=app-xxxxx   # WF1
DIFY_WF_DOC_REVIEW_KEY=app-xxxxx          # WF2
DIFY_WF_COPY_REVIEW_KEY=app-xxxxx         # WF3
DIFY_WF_COPY_GENERATE_KEY=app-xxxxx       # WF4
DIFY_WF_DOC_GENERATE_KEY=app-xxxxx        # WF5
DIFY_CHAT_APP_KEY=app-xxxxx               # APP1
```

## 2. 知识库设计

在 Dify Cloud 中需要创建 **4 个知识库**，与 RAGFlow 中的知识库对应：

### 2.1 知识库列表

| ID | 名称 | 内容 | 用于 |
|----|------|------|------|
| KB1 | 化妆品法规库 | 法规、标准、技术规范 | WF1、APP1 |
| KB2 | 体系文件模板库 | GMP模板、SOP范例、标准格式 | WF2 |
| KB3 | 文案合规案例库 | 违规案例、合规话术、禁用词表 | WF3、WF4 |
| KB4 | 企业知识库 | 企业内部文件、培训资料 | APP1 |

### 2.2 知识库配置参数

所有知识库统一配置：

| 参数 | 值 | 说明 |
|------|-----|------|
| 索引方式 | 高质量 | 使用 Embedding 模型 |
| Embedding 模型 | text-embedding-3-small | OpenAI 官方模型 |
| 检索模式 | 混合检索 | 向量 + 全文，取并集 |
| TopK | 5 | 每次检索返回 5 个分段 |
| Score 阈值 | 0.5 | 过滤低相关性结果 |
| Rerank | 开启 | 使用 Rerank 模型二次排序 |
| 分段长度 | 800 tokens | 法规文档建议中等长度 |
| 分段重叠 | 100 tokens | 避免上下文截断 |

### 2.3 需要上传的文档清单

**KB1 - 化妆品法规库（核心）：**
- 《化妆品监督管理条例》（2021）
- 《化妆品注册备案管理办法》
- 《化妆品安全技术规范》（2015版）
- 《化妆品标签管理办法》
- 《化妆品功效宣称评价规范》
- 《化妆品生产质量管理规范》
- 《已使用化妆品原料目录》
- 《化妆品禁用原料目录》
- 各地方监管通知、行业标准

**KB2 - 体系文件模板库：**
- GMP 体系文件标准模板
- 各类 SOP 模板（生产、质量、仓储等）
- 验证方案标准模板
- 检验规程标准格式

**KB3 - 文案合规案例库：**
- 化妆品广告违规案例汇编
- 化妆品宣称禁用词清单
- 各平台广告审核规则摘要
- 合规文案正面示例

**KB4 - 企业知识库：**
- 企业产品目录与配方摘要
- 内部培训资料
- 常见问题 FAQ

## 3. WF1 - 法规智能检索工作流

### 3.1 基本信息

| 属性 | 值 |
|------|-----|
| 应用类型 | Workflow |
| 名称 | 化妆品法规智能检索 |
| 描述 | 基于知识库检索，智能回答化妆品法规相关问题 |
| 调用方式 | POST /v1/workflows/run（blocking 模式） |

### 3.2 工作流节点设计

```
┌─────────┐     ┌──────────────┐     ┌───────────┐     ┌──────────┐
│  开始    │────▶│ 知识检索      │────▶│ LLM 节点   │────▶│  结束    │
│ (Start) │     │(KB1:法规库)   │     │(生成回答)   │     │ (End)   │
└─────────┘     └──────────────┘     └───────────┘     └──────────┘
```

### 3.3 节点详细配置

#### 节点1：开始节点 (Start)

**输入变量：**

| 变量名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| question | String | 是 | 用户的法规查询问题 |

> 对应前端调用：`inputs: { question: "烟酰胺使用限量要求" }`

#### 节点2：知识检索节点 (Knowledge Retrieval)

| 配置项 | 值 |
|--------|-----|
| 查询变量 | `{{#start.question#}}` |
| 知识库 | KB1（化妆品法规库） |
| 检索模式 | 混合检索（向量 + 全文） |
| TopK | 5 |
| Score 阈值 | 0.5 |
| Rerank 模型 | 开启 |

**输出变量：** `result`（Array[Object]，包含 content、score、title、metadata）

#### 节点3：LLM 节点

| 配置项 | 值 |
|--------|-----|
| 模型 | GPT-4o |
| Temperature | 0.3（法规问答需要准确性） |
| Max Tokens | 2000 |
| 上下文 | 引入知识检索节点的 `result` |

**System Prompt（系统提示词）：**

```
你是一位专业的化妆品法规咨询专家，精通中国化妆品相关法律法规、行业标准和技术规范。

## 角色定位
- 你是化妆品行业的法规顾问，服务对象是化妆品企业的法规、研发和质量人员
- 你需要基于提供的知识库内容，准确、专业地回答用户的法规问题

## 回答规范

### 必须遵守：
1. **严格基于知识库**：只使用检索到的知识库内容回答问题，不编造法规条文
2. **标注法规来源**：每个关键结论都要标注出自哪部法规、哪个条款
3. **使用 Markdown 格式**：用标题、列表、表格等结构化展示信息
4. **中文回答**：所有回答必须使用中文

### 回答结构：
1. **直接回答**：先用 1-2 句话给出核心结论
2. **详细解读**：分点展开具体法规要求
3. **注意事项**：补充实操中的注意点或常见误区
4. **法规依据**：列出引用的法规名称和条款编号

### 特殊情况处理：
- 如果知识库中没有相关信息，明确告知用户"当前知识库未收录该内容"，不要猜测
- 如果涉及多部法规，按优先级排列（条例 > 办法 > 规范 > 标准 > 通知）
- 如果法规有修订历史，标注当前有效版本

## 知识库检索结果
{{#context#}}
```

**User Prompt（用户提示词）：**

```
{{#start.question#}}
```

#### 节点4：结束节点 (End)

**输出变量：**

| 变量名 | 类型 | 映射 |
|--------|------|------|
| result | String | `{{#llm.text#}}` |
| sources | Array[String] | `{{#knowledge_retrieval.result#}}` |

> 前端通过 `extractAnswer(outputs)` 提取 `result` 字段
> 前端通过 `extractSources(outputs)` 提取 `sources` 字段

## 4. WF2 - 体系文件审核工作流

### 4.1 基本信息

| 属性 | 值 |
|------|-----|
| 应用类型 | Workflow |
| 名称 | 化妆品体系文件审核 |
| 描述 | 对照法规标准和模板，审核企业体系文件的合规性和完整性 |
| 调用方式 | POST /v1/workflows/run（blocking 模式） |

### 4.2 工作流节点设计

```
                                 ┌──────────────┐
                          ┌─────▶│ 知识检索-法规  │─────┐
┌─────────┐   ┌────────┐ │      │ (KB1)        │     │   ┌────────────┐   ┌──────────┐
│  开始    │──▶│问题分类 │─┤      └──────────────┘     ├──▶│ LLM 节点    │──▶│  结束    │
│ (Start) │   │(分类器) │ │      ┌──────────────┐     │   │(生成审核报告)│   │ (End)   │
└─────────┘   └────────┘ └─────▶│ 知识检索-模板  │─────┘   └────────────┘   └──────────┘
                                │ (KB2)        │
                                └──────────────┘
```

### 4.3 节点详细配置

#### 节点1：开始节点 (Start)

**输入变量：**

| 变量名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| document_content | String | 是 | 文档正文内容 |
| document_type | String | 是 | 文件类型（GMP体系文件/质量手册/SOP等） |
| file_name | String | 是 | 文件名称 |
| review_focus | String | 否 | 审核重点（用户自定义） |

> 对应前端：`inputs: { document_content, document_type, file_name, review_focus }`

#### 节点2：知识检索节点 - 法规标准 (Knowledge Retrieval 1)

| 配置项 | 值 |
|--------|-----|
| 查询变量 | `{{#start.document_type#}} 合规要求 审核要点` |
| 知识库 | KB1（化妆品法规库） |
| TopK | 5 |
| Score 阈值 | 0.4 |

#### 节点3：知识检索节点 - 模板对照 (Knowledge Retrieval 2)

| 配置项 | 值 |
|--------|-----|
| 查询变量 | `{{#start.document_type#}} 标准模板 格式要求` |
| 知识库 | KB2（体系文件模板库） |
| TopK | 3 |
| Score 阈值 | 0.4 |

#### 节点4：LLM 节点

| 配置项 | 值 |
|--------|-----|
| 模型 | GPT-4o |
| Temperature | 0.2（审核需要严谨） |
| Max Tokens | 4000（报告较长） |
| 上下文 | 引入两个知识检索节点的结果 |

**System Prompt：**

```
你是一位资深的化妆品行业质量管理体系审核专家，具有以下专业背景：
- 精通《化妆品生产质量管理规范》(GMP)
- 熟悉 ISO 22716 化妆品良好生产规范
- 了解 NMPA 对化妆品企业的监管要求
- 具备丰富的体系文件审核实战经验

## 审核任务
你需要对用户提交的化妆品体系文件进行全面审核，对照法规标准和模板，找出问题并给出改进建议。

## 审核维度（按优先级）

### 1. 法规合规性（最高优先级）
- 是否符合《化妆品监督管理条例》要求
- 是否满足 GMP 规范的具体条款
- 关键控制点是否覆盖完整

### 2. 内容完整性
- 必要章节是否齐全（对照标准模板）
- 关键要素是否遗漏（责任人、流程图、记录表单等）
- 引用的标准和法规是否为最新有效版本

### 3. 格式规范性
- 文件编号、版本号、生效日期是否规范
- 格式是否符合标准模板要求
- 术语使用是否准确统一

### 4. 可操作性
- 操作步骤是否具体可执行
- 判定标准是否明确量化
- 异常处理流程是否完善

## 输出格式要求（Markdown）

请严格按照以下格式输出审核报告：

```markdown
# 体系文件审核报告

## 一、文件基本信息
- 文件名称：{file_name}
- 文件类型：{document_type}
- 审核时间：{当前时间}

## 二、审核总结
{1-2句话概括审核结论，明确通过/需修改/不通过}

## 三、问题清单

### 🔴 高风险问题（必须整改）
{逐条列出法规合规性问题，每条包含：}
- **问题描述**：具体问题是什么
- **所在位置**：文档中的哪个章节/段落
- **法规依据**：违反了哪条法规要求
- **整改建议**：应该如何修改

### 🟡 中风险问题（建议整改）
{逐条列出内容完整性和格式规范性问题}

### 🟢 优化建议（可选改进）
{逐条列出可操作性和其他优化建议}

## 四、合规亮点
{列出文档做得好的方面，给予正向反馈}

## 五、整改优先级建议
{按紧急程度排列需要修改的事项}
```

## 审核原则
1. **客观严谨**：问题必须有法规依据，不凭主观判断
2. **分级标注**：严格区分高/中/低风险等级
3. **建设性**：每个问题都附带具体可落地的整改建议
4. **正向激励**：发现文档的亮点也要指出

## 法规标准参考
{{#context#}}
```

## 5. WF3 - 文案合规审核工作流

### 5.1 基本信息

| 属性 | 值 |
|------|-----|
| 应用类型 | Workflow |
| 名称 | 化妆品文案合规审核 |
| 描述 | 检查化妆品营销文案是否违反广告法和化妆品法规 |
| 调用方式 | POST /v1/workflows/run（blocking 模式） |

### 5.2 工作流节点设计

```
┌─────────┐   ┌──────────────┐   ┌───────────┐   ┌──────────┐
│  开始    │──▶│ 知识检索      │──▶│ LLM 节点   │──▶│  结束    │
│ (Start) │   │(KB3:文案案例) │   │(合规审核)   │   │ (End)   │
└─────────┘   └──────────────┘   └───────────┘   └──────────┘
```

### 5.3 节点详细配置

#### 节点1：开始节点 (Start)

**输入变量：**

| 变量名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| text | String | 是 | 需要审核的文案内容 |
| copy_type | String | 是 | 产品类型（护肤品/彩妆/洗护用品等） |

> 对应前端：`inputs: { text: copyText, copy_type: productType }`

#### 节点2：知识检索节点 (Knowledge Retrieval)

| 配置项 | 值 |
|--------|-----|
| 查询变量 | `化妆品 {{#start.copy_type#}} 广告 宣称 违规 禁用词` |
| 知识库 | KB3（文案合规案例库） |
| TopK | 5 |
| Score 阈值 | 0.4 |

#### 节点3：LLM 节点

| 配置项 | 值 |
|--------|-----|
| 模型 | GPT-4o |
| Temperature | 0.2 |
| Max Tokens | 3000 |

**System Prompt：**

```
你是一位化妆品广告合规审核专家，精通以下法规：
- 《中华人民共和国广告法》
- 《化妆品监督管理条例》
- 《化妆品标签管理办法》
- 《化妆品功效宣称评价规范》
- 各电商平台广告审核规则

## 审核任务
对用户提交的化妆品营销文案进行逐句合规审核，识别所有可能违规的表述。

## 审核标准

### 绝对禁止（高风险）：
1. **绝对化用语**：最佳、第一、唯一、100%等极限词
2. **医疗宣称**：治疗、治愈、消炎、抗菌、药用等医学用语
3. **虚假承诺**：永久、根除、彻底消除等不可能的效果承诺
4. **功效夸大**：7天见效、立竿见影等缺乏科学依据的时效承诺
5. **禁用对比**：贬低竞品、与药品对比等

### 风险警告（中风险）：
1. **暗示医疗效果**：修复、再生、抗敏等容易误解的表述
2. **未经验证的功效**：未做功效评价就宣称特定功效
3. **误导性数据**：来源不明的百分比数据、未标注测试条件
4. **特殊成分宣称**：对原料功效过度宣传

### 建议优化（低风险）：
1. **表述模糊**：可以更精准的表述
2. **格式规范**：应标注的注意事项未标注
3. **最佳实践**：可以优化的表达方式

## 输出格式（Markdown）

```markdown
# 文案合规审核报告

## 审核概要
- **产品类型：** {copy_type}
- **文案长度：** {字数}
- **风险等级：** {高风险/中风险/低风险/合规通过}

## 逐句审核结果

### 🔴 高风险问题
| 序号 | 原文 | 问题类型 | 违规依据 | 修改建议 |
|------|------|---------|---------|---------|
| 1 | "7天见效美白" | 虚假承诺+功效夸大 | 广告法第28条 | "持续使用，助力提亮肤色" |

### 🟡 中风险问题

## 9. Dify Cloud 操作步骤

### 9.1 准备工作

1. 注册/登录 [cloud.dify.ai](https://cloud.dify.ai)
2. 在「设置」→「模型供应商」中配置 OpenAI API Key
3. 确认可用模型：GPT-4o、GPT-4o-mini、text-embedding-3-small

### 9.2 创建知识库（4 个）

对每个知识库：

1. 进入「知识库」页面 → 点击「创建知识库」
2. 设置名称和描述
3. 上传对应文档（参见 2.3 节清单）
4. 索引方式选择「高质量」
5. 选择 Embedding 模型：`text-embedding-3-small`
6. 分段设置：最大 800 tokens，重叠 100 tokens
7. 检索设置：混合检索，TopK=5，Score=0.5

### 9.3 创建 Workflow 应用（5 个）

对每个 Workflow（WF1-WF5）：

1. 进入「工作室」→「创建应用」→ 选择「工作流」
2. 设置名称和描述
3. 按照上述设计文档添加节点（开始 → 知识检索 → LLM → 结束）
4. 复制提供的 System Prompt 和 User Prompt
5. 配置 LLM 参数（模型、Temperature、Max Tokens）
6. 连线并测试运行
7. 发布应用
8. 在「访问 API」页面获取 API Key（格式 `app-xxxxxxx`）

### 9.4 创建 Chatflow 应用（1 个）

1. 进入「工作室」→「创建应用」→ 选择「聊天助手」→ 选择「工作流编排」
2. 添加问题分类器节点，配置 3 个分类
3. 分别连接知识检索节点（KB1、KB4）和直接回答 LLM
4. 配置主 LLM 节点，**开启 Memory**
5. 添加 Answer 节点
6. 连线并测试对话（验证多轮对话记忆）
7. 发布应用
8. 获取 API Key

### 9.5 配置环境变量

获取 6 个 API Key 后，更新项目 `.env` 文件：

```env
# Dify Cloud

DIFY_WF_COPY_REVIEW_KEY=app-xxxxxx          # WF3
DIFY_WF_COPY_GENERATE_KEY=app-xxxxxx        # WF4
DIFY_WF_DOC_GENERATE_KEY=app-xxxxxx         # WF5

# Chatflow API Key
DIFY_CHAT_APP_KEY=app-xxxxxx                # APP1
```
